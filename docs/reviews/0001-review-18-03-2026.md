# Полный архитектурный, алгоритмический и код-ревью `casseted-core`

## Контекст ревью

Ревью проведено с учетом текущего этапа проекта: это не завершенный продукт, а ранний, но уже очень осмысленный фундамент для still-image GPU pipeline с формализованной VHS / analog v1 моделью.

Важно: в этом отчете я не считаю дефектами те вещи, которые явно отложены архитектурой текущего этапа:

- отсутствие видео и временного состояния
- отсутствие render graph
- отсутствие plugin / preset / SDK слоя
- отсутствие полной deck-accurate эмуляции VHS тракта
- отсутствие агрессивного runtime-кэширования как конечного решения

Ниже разделены:

- реальные риски и дефекты текущей реализации
- осознанные ограничения текущего milestone
- рекомендации, которые помогут следующему этапу не превратить удачный прототип в трудноразвиваемую основу

## Что было просмотрено

- весь Rust workspace
- архитектурные документы в `docs/architecture/`
- decision records в `docs/decisions/`
- math reference в `docs/math/signal-model-v1-formulas.md`
- все WGSL-проходы в `shaders/passes/`
- тестовый контур и PNG fixture’ы

## Ограничение верификации

Полноценный запуск `cargo test` в текущем окружении не состоялся не из-за кода проекта, а из-за рассинхронизации toolchain:

- workspace требует `edition = "2024"` и `rust-version = "1.85"` в `Cargo.toml:15-17`
- при этом `rust-toolchain.toml:1-3` закрепляет только `stable`, без точной версии
- в текущем окружении `cargo --version` = `1.82.0`, что не поддерживает `edition2024`

Следствие: ниже выводы опираются на полный статический разбор кода и документации, но не на завершенный локальный прогон тестов.

## Краткий вывод

Проект находится в хорошем состоянии для своего этапа.

Сильные стороны:

- очень внятные crate boundary
- редкая для ранней стадии согласованность между кодом, архитектурой и math-документацией
- хороший выбор текущего масштаба: система не переабстрагирована
- сильная идея stage-oriented visual regression вместо преждевременного построения сложной test platform

Главные риски:

- в `StillImagePipeline` уже появился опасный конфликт между формальной моделью и preview-представлением
- сборка репозитория недостаточно воспроизводима из-за неполного pinning toolchain
- часть параметров формальной модели публично доступна, но либо неактивна, либо частично теряет смысл при проекции
- центральный crate `casseted-pipeline` уже начал превращаться в слишком крупный узел ответственности

## Основные находки

### 1. Критичный архитектурный риск: `StillImagePipeline` хранит два независимых источника истины

Серьезность: высокая

Затронутые места:

- `crates/casseted-pipeline/src/lib.rs:45-47`
- `crates/casseted-pipeline/src/lib.rs:58-66`
- `crates/casseted-pipeline/src/lib.rs:88`
- `crates/casseted-pipeline/src/lib.rs:398-519`

Суть:

`StillImagePipeline` одновременно хранит:

- `model: Option<VhsModel>`
- `signal: SignalSettings`

Причем оба поля публичные.

Конструктор `from_vhs_model()` один раз проецирует модель в `signal`, но дальше между ними нет защищенного механизма синхронизации. Если вызывающий код меняет `pipeline.model`, то большая часть preview-зависимых параметров продолжит жить из старого `signal`, а часть model-only параметров уже начнет браться из нового `model`.

Почему это опасно:

- получается частично обновленный, частично устаревший pipeline
- поведение становится неочевидным: изменения формальной модели применяются не полностью
- ошибка будет не падением, а тихим неверным результатом, что хуже для графического/визуального кода

Пример класса ошибок:

- пользователь изменил `model.luma.bandwidth_mhz`
- новый projection ожидается в `signal.luma.blur_px`
- но `effective_preview_signal()` считает старый `signal` “override”-значением и сохраняет его
- в итоге часть `VhsModel` реально влияет на проходы, а часть нет

Рекомендация:

- сделать `model` и `signal` приватными
- ввести явную модель владения состоянием:
  либо `StillImagePipeline::from_vhs_model(model)` + `preview_overrides`
  либо `StillImagePipelineConfig { model, overrides }`
- добавить методы уровня `set_model()`, `set_signal_override()`, `clear_overrides()`, которые гарантированно пересчитывают projection
- убрать логику определения “override или не override” через голое сравнение двух публично мутируемых структур

### 2. Репозиторий не гарантирует воспроизводимую сборку

Серьезность: высокая

Затронутые места:

- `Cargo.toml:15-17`
- `rust-toolchain.toml:1-3`

Суть:

Workspace требует Rust `1.85` и `edition = "2024"`, но `rust-toolchain.toml` закрепляет только `stable`, а не конкретную совместимую версию.

Практический эффект уже проявился:

- в текущем окружении `stable` = `1.82.0`
- `cargo test` не стартует вообще

Почему это важно:

- новый contributor может получить нерабочий checkout
- CI и локальная разработка могут расходиться
- архитектурно хороший проект теряет воспроизводимость на самом базовом уровне

Рекомендация:

- закрепить точную версию toolchain, например `channel = "1.85.0"` или эквивалентную минимально поддерживаемую
- добавить CI job, который проверяет именно declared MSRV / required toolchain
- синхронизировать `justfile` и onboarding-документацию с этим требованием

### 3. Формальная модель допускает относительную chroma delay, но runtime-проекция отбрасывает отрицательные значения

Серьезность: средняя

Затронутые места:

- `crates/casseted-signal/src/vhs.rs:251-259`
- `crates/casseted-pipeline/src/lib.rs:333-355`

Суть:

`VhsChromaSettings.delay_us` задокументирован как относительная chroma delay против luma. Такая величина по смыслу естественно знаковая. Однако `chroma_offset_from_delay()` делает `delay_us.max(0.0)`, то есть отрицательные значения silently collapse в `0`.

Почему это проблема:

- формальная модель обещает большую выразительность, чем реально реализовано
- половина пространства параметра оказывается недоступной именно через `VhsModel`
- ручной preview-path отрицательные offset’ы поддерживает, а model-path нет

Это создает неприятную асимметрию:

- manual path может давать signed chroma misregistration
- formal model path теряет знак еще до шейдеров

Рекомендация:

- убрать `max(0.0)` в `chroma_offset_from_delay()`
- если нужны guardrail’ы, применять их позже и симметрично через signed normalization
- явно отметить в docs, какие поля formal model уже fully-active, а какие пока projected subset

### 4. Runtime создается заново на каждый вызов и уже сейчас является архитектурным bottleneck

Серьезность: средняя

Затронутые места:

- `crates/casseted-pipeline/src/lib.rs:73-197`

Суть:

На каждый `process_blocking()` / `process_with_gpu()` создаются или заново подготавливаются:

- `GpuContext` в blocking path
- bind group layouts
- bind groups
- render pipelines
- sampler
- набор промежуточных texture
- readback buffer

Для текущего milestone это допустимо как временное решение, но архитектурно именно здесь находится будущий предел масштабирования.

Почему это важно:

- still-image CLI это терпит, video path не переживет
- даже для серии still-кадров будет большой лишний overhead
- чем позже будет вводиться runtime cache / compiled pipeline object, тем дороже станет рефакторинг

Рекомендация:

- отделить описание pipeline от compiled runtime
- ввести что-то вроде `StillImageRuntime` / `CompiledStillPipeline`
- кэшировать:
  `wgpu::RenderPipeline`, `BindGroupLayout`, sampler, а позже и texture pool
- оставить текущий high-level API, но поверх переиспользуемого runtime-объекта

### 5. `casseted-pipeline` уже перегружен ролями и почти готов стать “god module”

Серьезность: средняя

Затронутое место:

- `crates/casseted-pipeline/src/lib.rs` целиком

Суть:

Один файл на 1434 строки сейчас содержит:

- публичный API pipeline
- projection из `VhsModel` в preview controls
- preview guardrail normalization
- uniform packing
- GPU resource setup
- render-pass orchestration
- readback logic
- часть knowledge о runtime layout
- inline unit tests

Это пока еще читаемо, но уже находится на грани точки, после которой каждое следующее изменение начнет цеплять слишком много разных слоев сразу.

Почему это важно:

- высокий риск merge conflicts
- сложно локализовать ответственность
- сложнее вводить отдельно алгоритмические и runtime-оптимизации
- будущий video / temporal слой почти наверняка потребует раскола этого модуля

Рекомендация:

На следующем шаге разделить хотя бы на:

- `projection.rs`
- `guardrails.rs`
- `uniforms.rs`
- `runtime.rs`
- `readback.rs`

Это даст проекту второй дыхательный запас без переписывания архитектуры.

## Архитектурный разбор

### Что сделано очень удачно

- Разделение на `types / signal / shaderlib / gpu / pipeline / cli / testing` выбрано хорошо.
- `casseted-signal` действительно выглядит как domain contract, а не как отражение текущего uniform layout.
- Документация не врет о текущем состоянии проекта и не обещает несуществующую генерализованную архитектуру.
- Решение не вводить render graph преждевременно выглядит правильным.
- Наличие отдельного `casseted-testing` для reusable image helpers тоже оправдано.

### Где архитектура уже начинает давать нагрузку

- `casseted-pipeline` стал одновременно местом domain projection и местом runtime orchestration.
- Публичный API недостаточно защищает инварианты между model-level и preview-level состоянием.
- Formal model уже шире, чем реально активный runtime subset, но эта граница пока в основном живет в документации, а не в типах.

### Вывод по архитектуре

На уровне workspace архитектура сильная.

На уровне центрального pipeline crate архитектура пока еще жизнеспособна, но требует следующего шага не в сторону “больше абстракций”, а в сторону “лучше защищенные границы и меньше смешения ответственности”.

## Алгоритмический разбор

### Сильные стороны алгоритмического слоя

- Есть ясное разделение formal model и current implementation subset.
- Разбиение на `input/tone -> luma -> chroma -> reconstruction` очень удачное для текущей стадии.
- Luma- и chroma-ветки действительно различаются по характеру деградации, а не только по набору коэффициентов.
- Документирование формул и visual regression cases заметно повышает управляемость изменений.
- Шейдеры написаны в логике, читаемой через физические/сигнальные термины, а не только через “набор художественных фильтров”.

### Основные алгоритмические ограничения текущего этапа

Это не дефекты, а именно ограничения текущего этапа:

- обработка ведется в gamma-coded пространстве, а не в linear-light
- transport остаётся spatial still-frame approximation
- часть формальной модели пока не активна в runtime:
  `phase_error_deg`,
  `chroma_phase_noise_deg`,
  `head_switching_*`,
  `output_transfer`,
  часть input/temporal semantics

Для раннего milestone это приемлемо, но важно не потерять прозрачность этого факта для будущих пользователей API.

### Алгоритмические риски

- Большая часть mapping coefficients пока hard-coded прямо в `casseted-pipeline`.
- Эти коэффициенты хорошо задокументированы, но слабо инкапсулированы.
- Если дальше добавлять новые VHS-artifact proxy тем же стилем, центральный projection layer станет слишком хрупким и плохо обозримым.

### Вывод по алгоритмам

Алгоритмический фундамент хороший.

Это не “набор случайных визуальных трюков”, а уже вполне дисциплинированная signal-model-first система. Следующий риск не в качестве текущих формул, а в росте числа эвристик без выделения более явного projection layer API.

## Код-ревью

### Сильные стороны кода

- Нейминг в целом аккуратный и читаемый.
- Код хорошо поддерживает narrative из документации.
- В small crates нет лишней сложности.
- WGSL-проходы понятны и не перегружены магией.
- Есть хорошие тесты на invariants и stage behavior, а не только smoke tests.

### Что стоит улучшить в коде

#### 1. Сравнение float-значений как способа определить override слишком хрупкое

Затронутое место:

- `crates/casseted-pipeline/src/lib.rs:398-519`

Сейчас логика “это untouched projected value или user override?” основана на прямом `==` между float-полями. Пока это работает внутри контролируемого happy-path, но плохо масштабируется на:

- сериализацию / десериализацию
- UI roundtrip
- будущие preset systems
- частично вычисленные значения

Рекомендация:

- хранить explicit override mask / enum / metadata
- не пытаться выводить пользовательское намерение из сравнения float’ов

#### 2. Повторяющийся fullscreen triangle boilerplate в шейдерах стоит централизовать

Затронутые места:

- все файлы в `shaders/passes/`

Это низкий приоритет, но общий vertex boilerplate уже дублируется во всех проходах. Когда число pass’ов вырастет, лучше вынести общий include-шаблон или shaderlib helper.

#### 3. В `casseted-testing` и CLI есть дублирование PNG I/O логики

Затронутые места:

- `crates/casseted-cli/src/main.rs`
- `crates/casseted-testing/src/lib.rs`

Пока это не проблема, но с ростом сценариев чтения/сохранения изображений имеет смысл выделить маленький shared helper crate или модуль для image I/O policy.

## Тестирование и документация

### Что хорошо

- Документация unusually strong для такого раннего этапа.
- Decision records осмысленные, а не формальные.
- Visual regression fixture strategy для текущего масштаба выбрана правильно.
- Есть stage-oriented tests, а не только one-shot end-to-end smoke tests.

### Что нужно сделать дальше

- восстановить воспроизводимую сборку через точный toolchain pinning
- добавить CI, который хотя бы проверяет:
  `cargo check`,
  `cargo test`,
  возможно `cargo fmt --check`
- при появлении runtime cache добавить отдельные tests на resource reuse / API invariants

## Что не является дефектом на текущем этапе

Я сознательно не записываю в “проблемы”:

- отсутствие temporal state
- отсутствие video pipeline
- отсутствие render graph
- отсутствие plugin API
- отсутствие полной симуляции head switching и chroma phase path

Это выглядит как корректно отложенный scope, а не как архитектурная ошибка.

## Рекомендуемый порядок следующих шагов

### Приоритет 1

- Закрыть проблему с toolchain reproducibility.
- Сделать `StillImagePipeline` инвариантно-консистентным и убрать два публичных mutable source of truth.

### Приоритет 2

- Разделить `casseted-pipeline/src/lib.rs` на projection / guardrails / runtime / uniforms.
- Сделать explicit representation active subset formal model.

### Приоритет 3

- Ввести compiled runtime / cache layer для переиспользования GPU pipeline state.
- Подготовить архитектуру к серии кадров, не обязательно сразу к полноценному video.

### Приоритет 4

- Расширять algorithmic subset formal model дальше, но уже поверх защищенного API и менее монолитного pipeline crate.

## Итоговая оценка

Если оценивать проект как ранний foundation milestone, а не как finished product:

- архитектура workspace: сильная
- алгоритмическая база: сильная
- качество документации: очень сильное
- зрелость runtime API: средняя
- готовность к следующему масштабу без рефакторинга: средняя-ниже средней

Итог:

Проект развивается в правильную сторону. Главная задача следующего этапа не “добавить больше эффектов”, а защитить уже удачную архитектурную идею от двух вещей:

- тихой рассинхронизации model-level и preview-level состояния
- естественного расползания `casseted-pipeline` в единственную точку всей логики системы

Если эти два риска закрыть сейчас, дальше расширять signal model и runtime будет значительно проще.
